## Assumptions tested

## Findings

### PR-01-未修复-[高]-Finality eligibility incorrectly depends on the writer being daily-job
- **位置**: Contract decision 1; S1 existing-row state machine.
- **问题类型**: 状态机漏洞 / 架构边界 / 不可直接实施
- **当前写法**: A row qualifies for `skipped_existing_nav` only when `status=final`, `writer=daily-nav-job`, and `nav_date` matches.
- **反例/失败场景**: A validated repair or rollback can intentionally produce the authoritative final row for a date. If its provenance is `nav-repair`, the next scheduled idempotency check blocks it forever as `existing_nav_not_final`, even though the lifecycle state is explicitly final and validated.
- **为什么有问题**: Lifecycle truth and provenance are different facts. Binding finality to one writer prevents legitimate recovery and forces operators either to lie about provenance or overwrite a correct row.
- **直接证据**: `src/maintenance/nav_history_repair/patch.py` is the existing validated, journaled, resumable, rollback-capable recovery path. `DailyNavJobService` may revisit the same prior business date on Monday after a Saturday finalization/repair.
- **影响**: Correct repaired rows become permanent blockers; operators may use unsafe overwrite or falsify writer metadata.
- **建议改法和验证点**: Eligibility should require supported finality version, `status=final`, matching `nav_date`, and no unresolved snapshot recovery. `writer` remains mandatory audit provenance but is not an eligibility discriminator. Only trusted application/maintenance code should construct final metadata; public request bodies must not accept arbitrary finality claims. Add a test that an explicitly final `nav-repair` row skips while a `status=maintenance` repair row blocks.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 高

### PR-02-未修复-[高]-Snapshot completeness predicate is underspecified and contradicts current successful-write state
- **位置**: Contract decision 1 and 2; S1 tests.
- **问题类型**: 契约缺失 / 状态机漏洞 / 测试缺口
- **当前写法**: The plan says only rows with “complete snapshot state” qualify, but does not define the exact persisted predicate.
- **反例/失败场景**: On a successful NAV + snapshot write, `NavRecordService` does not patch `details.snapshot_status=complete`; that complete object is constructed only for a later compensation target. Requiring an explicit complete marker would classify every normal successful row as non-final. Conversely, ignoring failure details would wrongly skip a partial write.
- **为什么有问题**: The implementation agent would have to redesign the state transition or guess the predicate, and tests could accidentally codify the wrong branch.
- **直接证据**: `src/app/nav_record_service.py` writes NAV first, persists snapshots second, and patches details only on snapshot failure. `DailyNavJobService._existing_nav_item()` currently treats `snapshot_persisted is False`, `snapshot_status == failed`, or an unresolved compensation task as recovery-required; absence of those markers is the successful steady state.
- **影响**: All new final rows may block, or partial rows may be skipped.
- **建议改法和验证点**: Define eligibility as recovery-first: any persisted failure marker or unresolved compensation task returns `recovery_required`; otherwise a valid finality block is eligible. Do not require a new `snapshot_status=complete` migration in this work unit. Add tests for (a) finality + no snapshot fields, (b) finality + failed fields, and (c) finality + unresolved local task.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 高

### PR-03-未修复-[中]-Metadata propagation is overcoupled across public and domain signatures
- **位置**: Contract decision 5; S1 primary files.
- **问题类型**: 过度耦合 / 架构边界 / 最佳实践偏离
- **当前写法**: Add four optional metadata parameters (`finality_status`, `writer`, `write_reason`, `valuation_as_of`) through account/application/portfolio/NAV layers.
- **反例/失败场景**: Every future field or invariant requires synchronized signature changes across `PortfolioService`, compatibility APIs, daily services, `PortfolioManager`, and `NavRecordService`. A caller can provide inconsistent combinations such as `status=final`, `writer=daily-report`, missing reason, or mismatched date.
- **为什么有问题**: The finality block is one validated value object, not four independent options. Spreading its fields across layers makes invalid states representable and increases public contract surface.
- **直接证据**: Existing call chains already repeat `overwrite_existing`, `dry_run`, `confirm`, `use_bulk_persist`, and `run_id` through several facades. The work unit exists partly because semantic ownership is already diffuse.
- **影响**: Signature churn, inconsistent metadata, compatibility mistakes, and difficult review.
- **建议改法和验证点**: Use one narrow internal immutable value (`NavWriteContext` or equivalent) owned by `src/app/nav_finality.py`, with validation and explicit factory/helper construction. Public HTTP/CLI requests must not accept arbitrary context fields. Pass one optional internal context only where needed; `NavRecordService` serializes it into details. Test invalid/mismatched contexts at the helper boundary.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 中

## Lens conclusions

- **Architecture boundary**: The repository-owned mutation lock is appropriate; finality eligibility must remain an application lifecycle concern.
- **Best practice**: Versioned metadata and fail-closed legacy handling are sound once snapshot and repair transitions are explicit.
- **Optimal solution**: Existing `details` storage plus one internal validated context is safer and smaller than a Feishu schema migration or general workflow engine.
- **Overengineering**: No new persistence entity or distributed lock is warranted.
- **Overcoupling**: Four independent propagated metadata parameters should be replaced by one validated internal context.
