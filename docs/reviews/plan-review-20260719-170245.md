# Plan Review

- Reviewed target: `docs/gateflow/repo-review-27/implementation-plan.md`
- Work unit: `repo-review-27 correctness hardening`
- Review mode: adversarial Gateflow plan review
- Scope: all seven slices, 27 finding mappings, state machines, contracts, tests, sequencing, residual risks
- Conclusion: `fail`

## Assumptions tested

1. Same-host locks plus absolute target compensation are sufficient for retry safety.
2. One compensation task/ID can represent every partial financial write.
3. NAV snapshot failure metadata will be discoverable on the already-persisted NAV row.
4. Removing only the outer valuation daemon thread is enough to satisfy end-to-end deadline/cancellation.
5. The Feishu typed-schema slice is code-generation-ready.
6. Futu legacy ETF rows and filtered fund rows can be distinguished without broadening zero-reconciliation.
7. App-level client-address enforcement remains effective when wrapper scripts are bypassed.
8. Historical rows and legacy compensation records remain readable without unsafe automatic mutation.

## Findings

### 01-未修复-[严重]-绝对目标补偿缺少前置状态校验，会覆盖补偿任务之后的合法交易
- **位置**: `Shared contracts and invariants / Compensation state machine`; S2 target operations
- **问题类型**: 并发恢复风险 / 状态机漏洞
- **当前写法**: retry 在 task lock 下把 holding/cash/snapshot 写成捕获的绝对 target；补偿 retry 明确“不拿 account lock”。
- **反例/失败场景**: 买入部分失败后产生旧 target；随后账户发生一笔合法交易；operator 再 retry 旧任务，旧 target 覆盖新交易后的持仓或现金。
- **为什么有问题**: target-state 只解决 crash-after-side-effect 的重复执行，不自动解决 stale task 对新状态的覆盖。task lock 只能防同一任务并发，不能与账户写入协调。
- **直接证据**: plan 规定 `Compensation retry only takes its task lock and never an account lock`，同时 operation 直接 `TARGET_SET`；当前账户写入由 `TradeService` 独立 account lock 保护。
- **影响**: 数据损坏、合法后续交易被回滚、恢复不可审计。
- **建议改法和验证点**: 每个 target payload 同时持久化 `before` 和 `target`；retry 固定按 `account lock -> task lock` 获取锁，并逐目标执行 compare-and-set：current==target 视为已完成，current==before 才允许写 target，其它状态进入 `CONFLICT`/`FAILED` 且不写。测试后续交易、混合已应用状态、crash 后 current==target。
- **修复风险（低/中/高）**: 高
- **严重程度（低/中/高/严重）**: 严重

### 02-未修复-[高]-多步骤金融 partial 没有原子恢复单元，单 task_id 契约不足以描述未完成目标
- **位置**: `Partial write contract`; S2 allowed changes 3-7
- **问题类型**: 契约缺失 / 不可直接实施
- **当前写法**: `PartialWriteError` 只有一个 `task_id`；operation 限定为单一 holding/cash/delete target。
- **反例/失败场景**: ledger 成功，holding 写失败后 cash 也未执行；或现金从 CASH/MMF 分拆扣款时第一行成功第二行失败。最终正确状态需要多个 target，但计划没有规定是停止、继续、创建多个任务、还是一个 task 内按何顺序恢复。
- **为什么有问题**: 实施 Agent 必须现场发明 recovery grouping、错误返回和 crash ordering，计划不满足 code-generation-ready。
- **直接证据**: 当前 `TradeService` 在 holding 失败后继续现金；`CashService.deduct_cash()` 可连续改 CASH 和 MMF。plan 的四种 operation 都是单类 target，exception 字段为 singular `task_id`。
- **影响**: 返回信息不完整、遗漏补偿、重复或乱序恢复。
- **建议改法和验证点**: 一个 compensation task 必须承载按固定顺序的 `targets[]`，每个 target 仍只允许现有四种窄类型；`PartialWriteError` 返回一个 task_id 指向完整未收敛目标集合。正常写前捕获完整 before/target；任一步失败后停止新副作用并把所有尚未确认/需校验的 target 写入同一 task。逐 target CAS、进度和最终状态都在同一任务事件中记录。
- **修复风险（低/中/高）**: 高
- **严重程度（低/中/高/严重）**: 高

### 03-未修复-[高]-NAV snapshot failure 的 marker/task_id 没有持久化闭环，existing-NAV recovery 可能永远不可发现
- **位置**: `NAV recovery contract`; S2/S4
- **问题类型**: 状态机漏洞 / 历史兼容
- **当前写法**: 计划要求把 failure marker 和 task_id 写入 NAV details，然后 existing NAV 读取 details 决定 recovery。
- **反例/失败场景**: 当前代码先写 NAV，再在 snapshot exception 后只修改内存 `nav_record.details`；没有第二次 NAV update。若补偿本地记录成功但 NAV details patch 失败，下一次 daily job 看到普通 existing row并跳过。
- **为什么有问题**: plan 没有指定负责 patch persisted NAV details 的 repository method、失败顺序或 fallback discovery source。
- **直接证据**: `src/app/nav_record_service.py:193-230` 在 snapshot failure 后只 `_mark_snapshot_failure(nav_record)` 和 `_record_compensation()`，没有持久化更新；`DailyNavJobService._existing_nav_item()` 只读 NAV row。
- **影响**: partial 被伪装为 completed-existing，恢复任务成为孤儿，回执/调度状态错误。
- **建议改法和验证点**: compensation 本地任务先 durable；再 best-effort patch NAV details 为 failed/task_id/digest。existing-NAV detection 同时查 row details 和本地 unresolved task by `related_record_id`，任一命中都返回 recovery_required。retry 成功后先确认 snapshot target，再 patch NAV details complete，再写 RESOLVED；details patch 失败保持 task FAILED/RUNNING 可重试。覆盖 details patch failure 和旧 row 无 marker 但有 task 的测试。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 04-未修复-[高]-价格计划没有消除内部不可取消线程，仍无法证明真实端到端 deadline
- **位置**: `Pricing deadline contract`; S5 allowed changes 6; deadline assertions
- **问题类型**: 并发恢复风险 / 测试缺口
- **当前写法**: 移除 ValuationService 外层 daemon thread，并把 deadline 传进 batch/provider/FX；但仍允许现有 BatchPricePlanner/FX ThreadPoolExecutor 并发。
- **反例/失败场景**: `as_completed(..., timeout=...)` 超时后离开 `with ThreadPoolExecutor` 会等待未完成 future；或 `shutdown(wait=False)` 返回但 worker 仍在后台做网络工作。
- **为什么有问题**: 计划声称“timed-out work does not continue”，却没有规定删除/限制所有线程池路径；现有 `src/pricing/batch.py` 和 `src/pricing/fx.py` 都创建 executors。
- **直接证据**: 当前 batch 通过 `ThreadPoolExecutor`/`as_completed`，FX 同样并发。仅传 deadline 不保证 Python thread 可取消。
- **影响**: 请求超过预算、后台继续刷新缓存/发网络请求、测试假通过。
- **建议改法和验证点**: deadline-controlled valuation path 必须不创建 ownerless worker：优先使用有单次 HTTP timeout 的原生 batch provider；剩余单标的和 FX 顺序执行并在每次调用/退避前检查 remaining time。并发只允许在所有 future 必定在 deadline 内由底层 timeout 收敛且 caller 等待全部结束的路径，不得用 `wait=False` 逃逸。测试枚举所有 provider/FX mocks 并断言返回后无活动工作/后续 cache write。
- **修复风险（低/中/高）**: 高
- **严重程度（低/中/高/严重）**: 高

### 05-未修复-[高]-S6 保留阻塞 open question，计划尚不能直接实施
- **位置**: S6 allowed change 8; `Risks and open questions`
- **问题类型**: open question 未收敛 / 不可直接实施
- **当前写法**: Feishu numeric type IDs 需要以后寻找 authoritative source，找不到时在 S6 停止询问。
- **反例/失败场景**: 前六个切片完成后因字段类型映射缺失停住；或 implementation 猜测 type ID。
- **为什么有问题**: Gateflow plan 必须在 accepted plan commit 前收敛 contract。该信息可在官方 Feishu/Lark field schema 文档中确认，不应后移。
- **直接证据**: 官方字段结构列出 `1=text, 2=number, 3=single select, 4=multi select, 5=date`，并提供 `ui_type`；当前项目文档只需要 text/select/number/date families。
- **影响**: sequencing 阻塞、schema check 错报/漏报。
- **建议改法和验证点**: 在计划中固定官方映射和兼容族：text=type1；number=type2；select=type3 or 4 only where docs say text/select；date/datetime=type5 (use `ui_type=DateTime` evidence, text remains accepted only where docs explicitly document text/datetime). 保存官方来源说明到 migrations docs；删除 blocking question。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 高

### 06-未修复-[中]-Futu legacy ETF 匹配和 zero-candidate 集合仍有歧义
- **位置**: S3 allowed changes 1-3
- **问题类型**: 契约缺失 / 历史兼容
- **当前写法**: 零 reconciliation 排除 `*_FUND`，但“incoming ETF 可保留 legacy fund enum metadata”。
- **反例/失败场景**: existing `US_FUND` SPY 与 incoming ETF SPY 应更新；existing `US_FUND` mutual fund absent from Futu snapshot must remain untouched。若只按 asset type 过滤，前者不匹配；若把全部 US_FUND 纳入 existing set，后者又被清零。
- **为什么有问题**: 计划没有规定先按 incoming eligible code join，再构造 absent zero candidates 的顺序。
- **直接证据**: 当前 `_build_position_diff()` 同时构造 eligible incoming 和 existing candidates；历史测试已有 legacy `US_FUND` SPY 更新场景。
- **影响**: 回归 finding 05 或破坏已有 ETF metadata behavior。
- **建议改法和验证点**: 先建立 eligible incoming by code；对每个 incoming code 从同 account/broker 全部 existing holdings 精确匹配并允许 legacy fund enum；完成 matched set 后，zero candidates 仅取未匹配的 stock/exchange_fund canonical types。测试 legacy ETF update + unrelated filtered fund untouched + absent canonical stock zeroed。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 中

## Architecture boundary review

- 正确方向：validation 在 TradeService、protocol validation 在 FeishuClient、loopback enforcement 在 ASGI app。
- 阻塞点：补偿 state ownership 尚未闭环；task lock、account lock 和 persisted NAV details 分属不同 owner，计划必须定义唯一推进顺序。

## Best-practice review

- append-only event + fsync、fail-closed transport、typed schema 和 absolute deadline 都符合该仓库风险等级。
- 但恢复必须 compare-and-set，超时必须避免不可取消 worker；否则只是形式上使用最佳实践。

## Optimal-solution review

- 不需要 SQLite/workflow engine。`before/target + CAS + account/task lock + append-only event` 是更小且足够安全的方案。
- 定价最优小方案是 deadline path 顺序/原生 batch，而不是尝试取消 Python network threads。

## Overengineering review

- repair journal 的 resume/rollback 是 finding 18 明确要求，不属过度设计。
- 不应再增加通用 operation registry；固定 `targets[]` union 足够。

## Overcoupling review

- S2 必须避免让 CompensationService 直接理解 TradeService 业务对象。task payload 使用序列化窄 target；apply handler 调用现有 storage/cash/snapshot repository boundary。
- S5 deadline 应由一个小 remaining-time helper 传递，不要把 ValuationService 依赖所有 provider implementation details。

## Open Questions

无。上述 findings 都可用仓库现有边界和已确认官方字段类型契约修订，无需用户决策。

## Residual Risks

- Cross-host Feishu uniqueness remains a documented infrastructure risk.
- Old delta-only compensation tasks remain manual-only.
- Explicit remote mode remains unauthenticated and operator-controlled.

These remain non-blocking only after the six plan findings are fixed and re-reviewed.
