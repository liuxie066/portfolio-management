## Findings

### S2-01-已修复-[高]-NAV snapshot compensation read model omitted the details required for CAS

- Entry point: `pm compensation retry --task-id ... --confirm` -> `CompensationService.retry()` -> `_apply_snapshot_target()` -> `storage.get_nav_on_date()`.
- Triggering input: a durable `HOLDINGS_SNAPSHOT_TARGET_SET` task after NAV persistence succeeded and snapshot persistence failed.
- Actual branch before fix: `get_nav_on_date()` returned an indexed `NAVHistory`; `NAV_INDEX_PROJECTION_FIELDS` omitted `details`, so `current_nav.details` became empty/absent.
- Expected behavior: compare the persisted original/failed details with the serialized before states, then batch-upsert snapshots and patch complete details.
- Actual behavior before fix: non-empty real NAV details could compare as `{}` and produce a false `state_conflict`, making the durable task non-executable.
- Direct evidence: `_apply_snapshot_target()` performs exact details CAS; `get_nav_on_date()` reads `_nav_objects` built from `NAV_INDEX_PROJECTION_FIELDS`; that projection previously omitted `details`.
- Impact: operators could not repair the exact NAV snapshot partial-write state S2 is intended to make recoverable.
- Fix: add `details` to the NAV index projection and protect the recovery read contract with a regression assertion.
- Fix risk: low; the field is already part of the NAV schema and is only added to the existing read projection.
- Severity: high.
- Final status: 已修复。

## Evidence and adversarial pass

- CLI retry requires `--confirm`; unsupported tasks do not execute.
- `PartialWriteError` distinguishes a durable recovery task from failure to persist recovery evidence.
