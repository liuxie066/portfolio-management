## Findings

### DR-S1-01-未修复-高-跨账户复用会绕过后续账户重新计算出的 stale-acceptance 策略
- **入口/函数**: `DailyNavJobService.run()` -> `ValuationService.calculate_valuation()` -> `RunQuotePool.fetch_batch()`
- **文件(行号)**: `src/app/valuation_service.py:52-85`; `src/app/run_quote_pool.py:127-134, 173-182`
- **输入场景**: 前一账户在市场关闭时以 `accept_stale_when_closed=True` 得到 `source=cache_fallback` / `is_stale=True` 的报价；账户顺序执行期间市场开盘，后一账户重新计算得到 `accept_stale_when_closed=False` 并请求同一 identity。
- **实际分支**: Pool 只按 `(canonical_code, market_type)` 判断 hit。`cached_before` 命中后，该 identity 不进入 misses，存量 stale payload 被直接标记 `is_from_run_pool=True` 返回。当前调用的 `accept_stale_when_closed=False` 位于 `fetch_kwargs`，只传给 miss 的底层 fetcher，从未用于决定 hit 是否可复用。
- **预期行为**: 后一账户当前的 stale policy 必须仍然有效；一旦当前调用不允许 stale fallback，该 stale run quote 不能命中，应失效并在剩余 attempt/deadline 内重新获取，失败则保持 missing 并阻断 NAV。
- **实际行为**: 后一账户接受了其自身 fetch policy 本应拒绝的过期报价；warning 虽保留 stale 标记，但估值仍使用该价格。
- **直接证据**: `ValuationService` 每次账户估值都会根据实时 market-open 状态计算 `accept_stale_when_closed`；`RunQuotePool` 的 hit/miss 条件只检查 identity 是否存在于 `_quotes`，没有读取 `fetch_kwargs` 或 payload 的 `source/is_stale`；结果循环无条件返回 `_quotes` 中的 payload。
- **影响**: 市场状态边界附近可能以过期价格完成并写入 NAV，违反“保持每账户 stale/fail-closed 策略”的已接受 contract；错误不是显式拒绝，而是带告警的成功估值，风险较高。
- **建议改法和验证点**: 在 pool owner 处增加当前调用的 reuse eligibility：`force_refresh=True` 不复用；`source=cache_fallback` 或 `is_stale=True` 只有当前 `accept_stale_when_closed=True` 才可复用。对不再 eligible 的存量 quote 失效并按同一 identity 的剩余 attempt/deadline 重新 delegate；若失败不得回落到刚失效的 quote。新增 closed->open 两次调用测试，断言第二次重新 fetch、失败时返回 missing/`failed_unique=1`，成功时覆盖为 fresh。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### DR-S1-02-未修复-中-Pool 吞掉 fetcher 异常使现有超时/异常诊断分支永久不可达
- **入口/函数**: `ValuationService.calculate_valuation()` -> `RunQuotePool.fetch_batch()`
- **文件(行号)**: `src/app/run_quote_pool.py:147-159`; `src/app/valuation_service.py:66-89`
- **输入场景**: 现有 `PriceFetcher.fetch_batch()` 在 deadline、provider adapter 或 unexpected runtime failure 上抛出 `TimeoutError` 或其它异常。
- **实际分支**: Pool 在 attempts 增加后用 `except Exception` 把异常转换成 `{}`。控制流正常返回 ValuationService，因此外层 `except TimeoutError` 和 `except Exception` 不会执行。
- **预期行为**: 引入 pool 不应删除既有 `价格获取超时（N秒）` / `价格获取异常: ...` 诊断；attempt state 可以在调用前推进，但异常仍应交给现有 valuation owner 解释。
- **实际行为**: 只剩每个 holding 的缺失/汇率告警，原始异常类型和消息被静默丢弃；对只含可 unit-price fallback 的 CNY cash/MMF 账户，甚至不会产生 blocking missing warning，导致 fetcher 故障在该账户结果中完全不可见。
- **直接证据**: Pool 的 broad catch 不记录也不返回错误；ValuationService 已有专门的异常格式化分支，但在使用 pool 时收不到异常。Focused tests只覆盖 empty mapping/invalid payload，没有 fetcher raise 场景。
- **影响**: 故障诊断信息回退，重复了本次事故中“内部异常不可恢复”的可观测性问题；operator 无法区分 provider 无结果、deadline 和代码异常。
- **建议改法和验证点**: 保留调用前的 attempt 增量，移除 broad catch并让异常回到 ValuationService；或使用显式结构化 error contract，但不要静默转空 mapping。新增 TimeoutError 和 RuntimeError 测试，断言 valuation warning 保留原有诊断、identity attempt 已计数且后续账户仍可按上限重试。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
