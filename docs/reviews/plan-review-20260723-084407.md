## Assumptions tested

- A `(canonical_code, market_type)` identity can be safely delegated through the current batch fetcher.
- A provider-boundary result can truthfully support the proposed `network_fetched` metric.
- Failure results are distinguishable from successful quote payloads without weakening valuation validation.
- Explicit per-run context avoids cross-run and concurrent-job mutable-state leakage.
- Optional argument propagation preserves non-daily and single-account compatibility.
- Existing blockers and Futu sync ordering still prevent premature quote requests.

## Required lenses

- **Architecture boundary review**: The run owns the mutable pool and passes it down explicitly; this keeps lifecycle authority out of the pricing singleton and persistent cache. The boundary is acceptable, subject to the batch canonicalization collision below.
- **Best-practice review**: Success-only admission, copy-on-return, bounded retries, and validation at both optimization and valuation boundaries are appropriate. The metric name must not claim evidence the boundary cannot observe.
- **Optimal-solution review**: Incremental sharing is safer and cheaper than an eager holdings-union prefetch because it preserves blockers, account ordering, and per-account deadlines.
- **Overengineering review**: One small pool is justified. A generic cache framework, durable store, provider rewrite, or new public configuration would be unjustified.
- **Overcoupling review**: Optional propagation across existing daily layers is narrow and one-directional. The pool must delegate through a callable/current fetcher contract rather than own or mutate the portfolio's fetcher.

## Findings

### PR-01-未修复-高-不同市场但相同 canonical code 的 miss 会被现有 batch planner 再次合并
- **位置**: `Contract decisions / Quote identity`、`Implementation slice S1`、cross-market test
- **问题类型**: 状态机漏洞 / 架构边界 / 不可直接实施
- **当前写法**: Plan 以 `(canonical_code, market_type)` 区分共享身份，并写明 miss 交给现有 `PriceFetcher.fetch_batch()`；但没有规定同一 delegated call 内如何隔离 canonical 相同、market 不同的两个 identity。
- **反例/失败场景**: 同一账户同时请求 `510300.US` 与 `510300.SH`，pool 正确得到两个 identity，却把两个 representative 一起交给现有 fetcher。现有 `BatchPricePlanner.fetch_batch()` 的 `original_by_canonical` 和 `primary_by_canonical` 只以 canonical 为键，两个请求被合并为一个，首个 raw code 决定市场，随后同一 payload 可能映射给两个市场。
- **为什么有问题**: Plan 承诺 cross-market 不共享，但下游 batch 边界会在 pool 之后重新消除 market 维度。仅测试两个顺序调用无法证明同一调用中的隔离，也会让实现 Agent 误以为 identity key 本身已经足够。
- **直接证据**: `src/pricing/batch.py::BatchPricePlanner.fetch_batch()` 使用 `original_by_canonical.setdefault(canonical, ...)` 和 `primary_by_canonical.setdefault(canonical, original)`，路由时只查看每个 canonical 的 primary；`_map_results_to_originals()` 再把一个 canonical payload 复制给所有 originals。
- **影响**: 跨市场同代码可能使用错误市场报价，造成无明显 missing warning 的错误估值；这比拒绝写入更危险。
- **建议改法和验证点**: Plan 必须规定 delegation partition：任何一次底层 `fetch_batch` 调用都不得包含 canonical 相同而 market 不同的 identity。为最小化常态行为变化，可将冲突 identity 拆成单独 delegated calls，非冲突 miss 仍维持一个批次并共用原 deadline。新增同一次 pool call 同时请求 `.US`/`.SH` 的测试，断言底层调用分离、payload 不交叉、attempt/metrics 分别计数。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### PR-02-未修复-中-`network_fetched` 在当前边界无法被真实测量
- **位置**: `Contract decisions / Observability`
- **问题类型**: 契约缺失 / 可观测性 / 最佳实践偏离
- **当前写法**: `network_fetched` 定义为“由 delegated provider call 获得有效报价的 distinct identities”。
- **反例/失败场景**: Pool 委托 `PriceFetcher.fetch_batch()` 后，结果可能来自持久缓存、fixed cash/MMF/crypto 构造或实时网络。Pool 只看到 payload，且一个 batch 内还可能混合多个来源。把所有 delegated success 统计为 network fetched 会把零网络缓存命中计成网络抓取。
- **为什么有问题**: 指标名称声称网络事实，但 pool boundary 没有请求级/provider 级证据。事故分析会据此错误判断 provider 调用是否已被消除。
- **直接证据**: `src/pricing/batch.py::BatchPricePlanner.fetch_batch()` 在同一方法中先走 `PriceCachePolicy.get()`，再处理 fixed factory，最后才进入 Tencent/US/provider；`PriceFetcher.fetch_batch()` 只返回 payload mapping，没有网络请求计数。
- **影响**: 回执/JSON 观测误导，无法可靠验证本修复是否减少真实 provider 请求，后续故障诊断可能做出错误结论。
- **建议改法和验证点**: 将字段改为边界可证明的 `fetcher_resolved`（delegated fetcher call 返回并通过校验的 distinct identities），另设 `fetch_attempted` 表示 delegated identities；不要在本 work unit 推导真实网络调用数。底层来源仍由 account realtime/cache/stale 分类表达。测试 persistent-cache/fixed payload 不被描述为网络抓取。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
