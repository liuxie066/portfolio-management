## Findings

### DR-S1-01-未修复-[高]-读取侧未按写入侧契约验证 finality，畸形或自相矛盾的 finality 可被当作最终 NAV
- **入口/函数**: `DailyNavJobService._existing_nav_item()` -> `evaluate_nav_finality()`
- **文件(行号)**: `src/app/nav_finality.py:45-55,94-118`; `src/app/daily_nav_job_service.py:134-146`
- **输入场景**: 已有行的 `details.finality` 手工或历史写成 `version=1,status=final,nav_date=目标日,writer=typo`，或写成语义冲突组合如 `status=final,writer=close-nav`；`valuation_as_of` 也可为非 ISO 垃圾值。
- **实际分支**: `evaluate_nav_finality()` 只检查 writer/reason 非空，不检查 writer 是否属于受支持集合、writer/status 组合是否属于可信写路径，也不验证非空 `valuation_as_of`。
- **预期行为**: 读取侧必须按同一版本化契约 fail closed；只有由 `NavWriteContext` 可合法构造的 finality 组合才能成为 `eligible_final`，其中 `final + daily-nav-job` 和 `final + nav-repair` 合法，其他矛盾组合或畸形字段应阻断。
- **实际行为**: 任何非空 writer/reason 都可让行进入 `skipped_existing_nav`，重新引入“错误状态被当成最终状态”的同类风险。
- **直接证据**: 写入侧 `NavWriteContext.__post_init__` 仅分别验证集合，读取侧第 110-113 行只验证非空；第 118 行直接返回 eligible。
- **影响**: 错误状态；daily job 可能跳过不可信 NAV 行，违反本 work unit 的核心 fail-closed 成功信号。
- **建议改法和验证点**: 在一个共享契约验证函数中定义 writer/status 合法矩阵；读取侧校验受支持 writer、合法组合、必需字段和非空 valuation timestamp 的 ISO 格式。增加 unknown writer、冲突 writer/status、invalid valuation timestamp 均返回 `existing_nav_not_final` 的测试。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 高

### DR-S1-02-未修复-[中]-运行时字段冲突被静默吞掉，返回 run_id 与持久化 provenance 可能不一致
- **入口/函数**: `AccountNavRecorderService.record()` -> `NavWriteContext.with_runtime()` -> `NavRecordService.record_nav()`
- **文件(行号)**: `src/app/nav_finality.py:60-71`; `src/app/account_nav_recorder_service.py:115-128`; `src/app/nav_record_service.py:163-181`
- **输入场景**: 内部调用者传入 context 的 `run_id=A`，同时 recorder/service 参数解析出 `run_id=B`；valuation timestamp 也可能预填为 A 时点但 snapshot 实际为 B 时点。
- **实际分支**: `with_runtime()` 使用 `self.run_id or run_id` 和 `self.valuation_as_of or runtime_value`，静默保留 context 旧值。
- **预期行为**: 同一写入的 runtime facts 冲突应立即拒绝，不能让 API 结果/日志使用 B 而 `details.finality.run_id` 使用 A。
- **实际行为**: recorder 返回 `resolved_run_id=B`，持久化 details 可保留 A，审计关联失真且没有异常。
- **直接证据**: `AccountNavRecorderService` 把 `resolved_run_id` 放入 result/snapshot，但 `NavRecordService` 持久化的是 context 自带 run_id；`with_runtime()` 没有冲突检测。
- **影响**: 审计/追踪状态不一致，故障恢复和运行证据可能关联到错误 run。
- **建议改法和验证点**: `with_runtime()` 对两个非空且不同的 run_id/valuation_as_of raise `ValueError`；测试冲突 fail closed，并保留相同值或缺省填充的正常路径。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

### DR-S1-03-未修复-[中]-直接 Python facade 传入 datetime 会在新增 finality 日期校验处回归失败
- **入口/函数**: `PortfolioManager.record_nav()` / `NavRecordService.record_nav()`
- **文件(行号)**: `src/app/nav_record_service.py:70-86,163-176`; `src/portfolio.py:223-243`
- **输入场景**: 历史 Python 调用传 `nav_date=datetime(2026,3,19,...)`。`datetime` 是 `date` 子类，且旧计算链可使用其 `strftime` 等日期方法。
- **实际分支**: `today` 保留 datetime；`NavWriteContext` 把 nav_date 归一为 date；随后 `resolved_context.nav_date != today` 为真并 raise。
- **预期行为**: 在记录链入口统一把 datetime 归一为 date，再用于计算、context 和比较；或明确拒绝但需要公共契约变更。当前 S1 不应无意改变兼容行为。
- **实际行为**: 新 finality 一致性检查让此前可工作的输入失败。
- **直接证据**: `today = nav_date or bj_today()` 未归一化，而 `NavWriteContext.__post_init__` 调用 `_coerce_date()`。
- **影响**: Python 兼容入口回归；特定自动化调用可能无法写 NAV。
- **建议改法和验证点**: 在 `NavRecordService.record_nav()` 一开始将 datetime 转 date，并添加直接 facade datetime 测试。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
