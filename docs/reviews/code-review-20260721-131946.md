## Findings

### 1-未修复-高-本地 NAV 索引会丢失 finality，进程重启后合法 final 行被误判为未完成
- **入口/函数**: `DailyNavJobService.run()` -> `_existing_nav_item()` -> `storage.get_nav_on_date()` -> local NAV index reconstruction
- **文件(行号)**: `src/app/daily_nav_job_service.py:99-147`; `src/feishu/repositories/nav_history_repository.py:35-60, 246-296, 716-729`
- **输入场景**: 远端已存在由 canonical daily job 写入、带合法 `details.finality.status=final` 的目标日期记录；服务或定时任务进程重启，磁盘 local NAV index 已存在且包含该行。
- **实际分支**: `_ensure_nav_index_loaded()` 优先从磁盘缓存重建 `NAVHistory`，但缓存 payload 和重建对象都不保存 `details`。`_get_indexed_navs()` 看到非空列表后不会强制刷新远端。随后 `_existing_nav_item()` 得到 `details={}`，`evaluate_nav_finality()` 走 `missing_finality`，返回 `existing_nav_not_final`。
- **预期行为**: 合法 final 行在进程重启后仍应被识别为 `skipped_existing_nav`；duplicate audit 已读取远端时，existing-row decision 应基于同一份新鲜远端事实或至少保留完整 finality details。
- **实际行为**: canonical final 行被误报为需要人工分类，自动任务失败；只有内存中刚写入且尚未重启的进程可能正确跳过。
- **直接证据**: `_build_nav_index_payload()` 生成的 `nav_records` 没有 `details`；`_ensure_nav_index_loaded()` 构造 `NAVHistory(...)` 时也没有传 `details`；`_get_indexed_navs()` 对任何非空本地缓存直接返回；`evaluate_nav_finality()` 在缺少 `details.finality` 时明确返回 `missing_finality`。此外 `_audit_duplicates()` 虽然在同一 job 中先从 Feishu 读取了包含 `details` 的记录，却没有刷新 local/memory NAV index。
- **影响**: 核心 daily-job 幂等语义在常见的进程重启/定时启动场景下失效，原本应成功跳过的早晨任务会变成阻断失败；这直接破坏本 work unit 的主要成功信号。
- **建议改法和验证点**: 在 NAV index row 中持久化并重建 `details`；让 account-scoped duplicate audit 复用其已读取的远端 records 刷新同一 NAV index，避免 stale cache 决定 finality。新增回归测试：先构造只含磁盘缓存的 storage，再执行 account duplicate audit + `get_nav_on_date`，确认合法 finality 完整保留并被 daily job 判定为 `skipped_existing_nav`；同时确认 legacy row 仍为 `existing_nav_not_final`。
