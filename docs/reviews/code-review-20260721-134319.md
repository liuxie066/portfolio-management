### 2-未修复-高-Feishu 缺少 details 字段时 authoritative write 静默降级并谎报 finality 已持久化
- **入口/函数**: `DailyNavJobService.run()` -> `NavRecordService.record_nav()` -> `write_nav_record()` / `write_nav_records()` -> `_execute_single_nav_write()` / batch fallback
- **文件(行号)**: `src/feishu/repositories/nav_history_repository.py:397-452,540-583`; `src/app/nav_record_service.py` finality serialization path
- **输入场景**: canonical daily job 或其他 classified writer 生成带 `details.finality` 的 NAV；Feishu `nav_history` 表尚无或无法识别 `details` 字段，写请求返回 `FieldNameNotFound`。
- **实际分支**: 单行路径捕获异常后执行 `fallback_fields.pop('details', None)` 并重试成功；bulk update/create 路径同样删除 `details` 后重试。随后单行 `cache_row` 仍由原始 `nav.details` 构造，bulk fallback rows 也未删除缓存里的 `details`，因此调用方得到成功，本地缓存一度声称存在 finality，但远端 authority 实际没有 provenance。
- **预期行为**: 当前 work unit 把 `details.finality` 定义为 authoritative existing-row decision 的必要持久化契约。只要待写 NAV 携带 finality，`details` 无法持久化就必须 fail closed；不能把丢失 finality 的行当作成功写入。只有确实不携带 finality 的 legacy low-level write 才可保留兼容 fallback（若仍需要）。
- **实际行为**: 首次 canonical write 可返回成功但远端行没有 finality；同进程本地缓存甚至短暂制造“已 final”假象。下次 account audit/重启从远端刷新后才变成 `missing_finality` 并阻断，造成先成功、后失败的不一致状态。
- **直接证据**: `_execute_single_nav_write()` 的 `FieldNameNotFound` handler 无条件删除 `details` 后重试，并在第 439 行用原始 `nav` 创建 cache row；batch update/create fallback 在第 560/578 行删除远端 payload 的 `details`，但保留 `update_rows_for_cache` / `create_rows_for_cache` 中的 details。当前 tests 没有覆盖 classified NAV 遇到 `FieldNameNotFound` 必须失败。
- **影响**: 直接破坏 bug 2 的 finality/provenance 持久化成功信号，也使本地缓存与 Feishu authority 短暂分叉；早晨自动任务可能报告成功，下一次运行却因 legacy/non-final row 阻断。
- **建议改法和验证点**: 在 single 和 batch fallback 前识别待写 payload 是否包含 `details.finality`；若包含则抛出清晰错误并禁止无 details 重试/缓存更新。为 single create、single update、batch create/update 各覆盖至少一条 fail-closed 回归；同时保留一条无 finality legacy payload 的 fallback 兼容测试（若该兼容仍是明确需求）。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高
