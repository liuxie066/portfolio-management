## Findings

### 01-未修复-[高]-不同 journal 的同账户 NAV repair 可并发通过 CAS 并互相覆盖
- **入口/函数**: `patch.run()` -> `_apply_journal()` / `_rollback_journal()`
- **文件(行号)**: `src/maintenance/nav_history_repair/patch.py:572-634,637-694,831-847`
- **输入场景**: 两个进程同时对同一账户、同一 NAV 日期执行不同 patch；它们生成不同 digest/journal path。
- **实际分支**: 每个执行只锁 `nav-repair:<journal-path>`；两个不同 journal 的锁互不冲突。两边可在写入前都读取同一 original state 并通过 original/target 校验，然后各自用完整 `NAVHistory` 写回。
- **预期行为**: 同账户 NAV repair 的 live-state 校验、写入和 journal outcome 必须与其他 repair/compensation 写共享账户锁，保证 CAS 判断到写入之间不可被同机并发写穿透。
- **实际行为**: 不同 journal 可交错执行，后写者可能覆盖先写者的目标字段，而两个 journal 都记录 `applied`/`COMPLETED`。
- **直接证据**: `_apply_journal()` 和 `_rollback_journal()` 仅获取按 journal path 派生的 `process_lock`；仓库已有 `account_lock_key()`，但此路径未使用。
- **影响**: journal 与 live NAV 状态不一致、成功结果不可信、历史修复可丢失。
- **建议改法和验证点**: 按现有锁顺序在 journal lock 外层获取 `account_lock_key(context.account)`；测试 apply/rollback 均获取账户锁，并验证第二个 overlapping plan 在首个 plan 完成后因 state conflict fail closed。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 02-未修复-[高]-崩溃撕裂的最后一条 JSONL 会让 resume/rollback 永久不可读
- **入口/函数**: `_append_journal()` / `_read_journal()`
- **文件(行号)**: `src/maintenance/nav_history_repair/patch.py:435-486`
- **输入场景**: 进程在 append 最后一条 row/state event 时退出，文件尾只写入部分 JSON 且没有换行。
- **实际分支**: 读取时对每一行严格 `json.loads()`；最后一条 torn tail 直接 `SystemExit`。后续 append 仍从 torn bytes 后继续，无法自然恢复。
- **预期行为**: append-only crash journal 应接受并截断唯一允许的未换行 torn tail；中间损坏或带换行的 malformed event 仍应 fail closed。
- **实际行为**: 单次崩溃可使 journal、resume 和 rollback 全部不可用，违背 crash-safe recovery 目标。
- **直接证据**: `_append_journal()` 使用 append/fsync，但没有尾部修复；`_read_journal()` 对最后一个未换行 fragment 与中间损坏同样报错。
- **影响**: 部分已提交 NAV patch 失去自动恢复/回滚路径。
- **建议改法和验证点**: 二进制读取 journal；仅忽略最后一个未换行且不可解析的 fragment，并在下次 append 前在锁内 truncate 到最后一个完整换行；测试 partial apply 后手工追加 torn bytes，resume 仍完成且 journal 再次全量可解析。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 03-未修复-[中]-existing-NAV recovery 回执仍显示 unknown error
- **入口/函数**: `DailyNavJobService._existing_nav_item()` -> `NavHistoryReceiptService._item_lines()`
- **文件(行号)**: `src/app/daily_nav_job_service.py:112-130`; `src/app/nav_history_receipt_service.py:199-210`
- **输入场景**: 同日 NAV 已存在，details 或本地 compensation 指示 snapshot recovery required。
- **实际分支**: job item 只返回 `snapshot_error`，receipt 非 skipped 失败分支只读取 `item["error"]`。
- **预期行为**: 已保留的 `recovery_required` 状态应在飞书回执中携带实际 snapshot/compensation 错误，而不是丢失原因。
- **实际行为**: 回执显示 `recovery_required｜unknown error`；JSON 中虽有 `snapshot_error`，运营主入口仍缺失可执行原因。
- **直接证据**: recovery item 没有 `error`；receipt line 209 不读取 `snapshot_error`。
- **影响**: 运营无法从回执判断恢复原因，降低本 slice 试图恢复的 partial 可观测性。
- **建议改法和验证点**: recovery item 同时设置 canonical `error` 为 snapshot/compensation error；增加 receipt 回归测试，保留 `snapshot_error` 和 task/retry metadata。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
