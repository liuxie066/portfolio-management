### 2-已修复-高-FieldNameNotFound 会静默删除 authoritative finality 后返回成功
- **入口/函数**: `NavRecordService.record_nav()` -> `write_nav_record()` / `write_nav_records()` -> `_execute_single_nav_write()` / batch fallback
- **文件(行号)**: `src/feishu/repositories/nav_history_repository.py:397-473,582-629`; `tests/test_nav_bulk_upsert_minimal.py:285-458`
- **输入场景**: single create、single update、batch create 或 batch update 正在写入带 `details.finality` 的 NAV，Feishu 返回 `FieldNameNotFound`。
- **实际分支**: the shared guard inspects dictionary or serialized details payloads before the compatibility fallback. If any payload contains the `finality` key, it raises a clear fail-closed `RuntimeError` before the no-details retry and before local cache publication. Payloads without finality retain the legacy optional-details fallback.
- **预期行为**: authoritative finality either persists to Feishu or the write fails; local cache must not claim finality that the remote authority rejected.
- **实际行为**: four regressions prove single/bulk create/update each make exactly one failing Feishu call, never retry without details, and never publish a local cache row. A separate legacy regression proves the compatibility branch remains conditional rather than being removed globally.
- **直接证据**: `_fields_contain_finality()` parses serialized details and detects the finality key; `_fail_closed_if_finality_would_be_dropped()` raises before every single/batch `pop('details')` fallback; cache updates occur only after those guarded write blocks return successfully.
- **影响**: bug 2 的 persistence success signal 已闭环；canonical writes can no longer report success with remote finality missing or create a local-only final state.
- **建议改法和验证点**: 已按原建议修复；保留 single/bulk create/update fail-closed tests and the no-finality compatibility test.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高
