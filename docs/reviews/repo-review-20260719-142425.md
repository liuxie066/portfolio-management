### 01-未修复-[严重]-幂等重试只去重账本记录，仍会重复修改持仓或现金
- **入口/函数**: `TradeService.buy()` / `sell()` / `deposit()` / `withdraw()`
- **文件(行号)**: `src/feishu/repositories/transactions_repository.py:31-77`; `src/feishu/repositories/cash_flow_repository.py:29-53`; `src/app/trade_service.py:85-149,192-257,287-306,334-353`; `src/models.py:401-420`
- **输入场景**: 使用同一 `request_id` 重试买卖，或重放内容相同的出入金。
- **实际分支**: repository 检测到已有记录后返回旧 `record_id`；上层继续执行持仓增减和现金扣加。
- **预期行为**: 幂等重放应返回原结果，并跳过整个业务单元的所有副作用。
- **实际行为**: 账本只有一条记录，但持仓或现金可以被执行两次；未传 `request_id` 时又先生成随机 ID，默认调用也难以形成稳定业务幂等键。
- **直接证据**: `add_transaction()` / `add_cash_flow()` 的重复分支只返回模型；`TradeService` 未接收“新建/重放”状态，随后无条件调用 holdings/cash 更新。
- **影响**: 金融账本与持仓/现金静默分叉，重复买入扣现、重复卖出加现或重复出入金。
- **建议改法和验证点**: repository 返回结构化 `created/replayed` 结果，重放时短路整个业务流程；以并发和串行重试测试证明账本、持仓、现金均只变化一次。
- **修复风险（低/中/高）**: 高
- **严重程度（低/中/高/严重）**: 严重

### 02-未修复-[高]-卖出未校验可用数量，可生成负持仓或无持仓加现金

- **输入场景**: 持仓 1 卖出 2，或直接调用 manager/service 卖出不存在的持仓。
- **实际分支**: 入口只检查记录存在性或完全不检查；repository 直接执行 `quantity + quantity_change`，不存在时静默返回。

- **影响**: 凭空创造现金、负持仓和不可对账状态。
- **建议改法和验证点**: 增加权威的原子可用数量检查并让扣减失败阻断交易；测试超卖、broker 不匹配、并发卖出和零持仓。
