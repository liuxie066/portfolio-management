### 1-未修复-[中]-fallback safety default permits future write commands to replay when classification is omitted
- **入口/函数**: `scripts.pm._service_or_fallback()` and every future CLI command routed through it.
- **文件(行号)**: initial S7 implementation at `scripts/pm.py:258`; existing call sites at `scripts/pm.py:326-706`.
- **输入场景**: a future write-capable command calls `_service_or_fallback(args, via_service, direct)` but omits the new keyword.
- **实际分支**: `allow_fallback=True` is silently selected by the helper default, so `PortfolioServiceUnavailable` invokes the direct backend.
- **预期行为**: every service-backed command must explicitly declare whether fallback is safe; omission must fail during development instead of selecting the unsafe write-replay behavior.
- **实际行为**: current write commands were correctly marked false, but the helper contract allowed an unclassified future write command to regain the exact replay defect fixed by finding 15.
- **直接证据**: the initial signature was `_service_or_fallback(..., *, allow_fallback=True)` while read call sites relied on omission; no parser/type boundary forced command classification.
- **影响**: future maintenance could reintroduce duplicate financial writes and notifications after an ambiguous service failure.
- **建议改法和验证点**: remove the default, pass `allow_fallback=True` explicitly at every read-only call and `False` at every write-capable call, and structurally verify that no call omits the keyword.
