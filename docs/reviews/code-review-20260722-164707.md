## Findings

### 1-未修复-低-aware `expires_at` 未来时刻会被当作已过期并以 stale 标记返回（fail-open 方向标记错误）
- **入口/函数**: `PriceCachePolicy.get()` -> `_parse_expires_at()` -> `except Exception: is_expired = True` -> `_market_has_traded_since()` -> `has_market_session_between()`
- **文件(行号)**: `src/pricing/cache.py:38-45,112-117,121-127`; `src/market_time.py:228`
- **输入场景**: storage 返回的 `expires_at` 为带时区字符串（如 `2026-08-01T00:00:00+00:00`，未来时刻）。`_parse_expires_at` 经 `fromisoformat(value.replace("Z","+00:00"))` 产出 aware datetime；`expire_dt <= bj_now_naive()` 对 naive/aware 混合比较抛 `TypeError`，被 `except Exception: is_expired = True` 吞掉。
- **实际分支**: 条目实际未过期，却进入 expired 分支；语义路径下 `_market_has_traded_since(code, 未来aware时刻)` → `has_market_session_between` 中 `end_aware <= start_aware`（market_time.py:228）返回 `False`（"未交易过"），于是该新鲜缓存以 `stale=True`/`source=cache_fallback`/`cache_status=stale_fallback` 返回。
- **预期行为**: 未过期条目应走 `cache_status="hit"`；解析失败应 fail-closed（拒绝或按 hit 处理），不应把新鲜报价误标为 stale_fallback 进入 `价格汇总 stale_fallback=N` 告警口径。
- **实际行为**: 新鲜报价被误标为 stale（价格数值本身仍是新鲜的，估值值不错，但可观测性失真；若 NAV 汇总/回执以 stale_fallback 计数触发人工排查，会造成误导）。已实测复现：aware 过去时刻路径工作正常（`_as_aware` 兼容），仅未来 aware 时刻落入反向窗口。
- **直接证据**: 复现脚本确认 `TypeError: can't compare offset-naive and offset-aware datetimes` 被吞、`has_market_session_between(us, 未来aware, naive now)` 返回 `False`。当前生产 storage（`local_cache.py:134-138` 以 naive `strftime` 存储）不会产出 aware expires_at，故现网不可达；但 `_price_mixin`/其他 storage 实现或未来迁移可达。
- **影响**: 低。估值数值不受影响（fail-safe 方向），仅 stale 标记/统计失真；但语义路径上线后该边缘从"不可达"变为"可达"，属于本次改动激活的既有缺陷。
- **建议改法和验证点**: 在 `_parse_expires_at` 之后把 aware datetime 统一转为北京 naive（`astimezone(TZ_SHANGHAI).replace(tzinfo=None)`），或在比较前用 `MarketTimeUtil._as_aware` 对齐两侧；补一条 aware expires_at（过去/未来各一）的 `PriceCachePolicy.get` 回归测试。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低

### 2-未修复-低-`has_market_session_between(None, ...)` 返回 False（接受 stale），与函数自身 fail-closed 契约相反
- **入口/函数**: `MarketTimeUtil.has_market_session_between()`
- **文件(行号)**: `src/market_time.py:213-214`
- **输入场景**: 未来调用方直接传 `start=None` 或 `end=None`。
- **实际分支**: 函数对未知/fund 市场类型返回 True（拒绝，fail-closed），但对 None 输入返回 False（"无交易"，即接受 stale）——同一函数内两种"信息不足"输入给出相反的失败方向。
- **预期行为**: 信息不足应一律 fail-closed（返回 True 或抛错），与 docstring "未知保守返回 True" 一致。
- **实际行为**: 当前唯一生产调用方 `_market_has_traded_since`（cache.py:134-135）对 `expire_dt=None` 有前置守卫返回 True，故现网不可达；但这是一个 fail-open 方向的 API 陷阱，任何新调用方漏掉守卫即静默接受 stale。
- **直接证据**: `src/market_time.py:213-214` `if start is None or end is None: return False`；对比 222-223 行未知市场返回 True。
- **影响**: 低（当前不可达），但属 fail-open 方向的隐患，修复成本一行。
- **建议改法和验证点**: 改为 `return True`（或 `raise ValueError`），并补一条 None 输入的单元测试锁定方向。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低

## 已排除（验证后排除的疑点）

- **US 探针 DST/时区一致性**：排除。探针以 NY-local 9:30 构造（zoneinfo 直接赋 tzinfo 对 9:30 无歧义，DST 由 zoneinfo 正确处理），与 Shanghai-aware 的 start/end 比较是 aware-aware 绝对时间比较，一致。实测：冬季（EST）周六 06:00 BJ→周一 20:00 BJ = False（open 22:30 BJ），→周一 23:00 BJ = True；夏季（EDT）对应 21:30 BJ 边界同样正确；跨 2026-03-08 DST 切换窗口 = True。
- **`is_open(start)` 提前 True 与 CN/HK 午休**：排除。CN 午休起点 12:00 的窗口由同日 13:00 下午场探针捕获（有测试 `test_cn_lunch_start_still_catches_afternoon_session`）；HK 午休（12:00-13:00）经脚本实测同样正确；午间内的短窗口（12:30→12:45）正确返回 False。
- **`probe_dt > end_aware → return False` 提前退出**：排除。探针在日内按时间升序、跨日按日期升序，单调递增，提前退出不会漏报。
- **45 天扫描上限**：排除。`STALE_RETRIEVAL_WINDOW_SEC = 40 天 < 45 天`，经 cache 路径不可达上限耗尽分支；即使 40 天窗口扫描耗时实测约 0.005ms，性能非问题。
- **aware 过去时刻的 expires_at**：排除。`has_market_session_between` 经 `_as_aware` 保留 aware start、naive end 补 Shanghai，aware-aware 比较正常工作（仅未来时刻落入 finding 1 的反向窗口）。
- **`expires_at` 缺失/None 按新鲜 hit 处理**：排除，属既有行为且未改变（cache.py:112 的 `if getattr(...)` 守卫为既有代码；`_market_has_traded_since` 对 None 另有 fail-closed 守卫）。
- **`detect_market_type` 分类的 fail-open 方向**：排除。实测 FUTU→us、BRK.B→us、00700/0700→hk、000001/513500→cn、090010→fund、900901→fund（拒绝方向，可接受）；`CASH`→us 与 `BTC-CRYPTO-USD`→us 虽为误分类，但 cash/mmf/crypto 走 fixed_factory 路径（service.py:139-162、batch.py:93-108）在 `cache_policy.save` 之前返回，永不入缓存，语义检查不可达；且 crypto/CASH 价格为常量或实时换算，无实际危害。
- **显式窗口契约保持**：排除。`semantic_stale = accept_stale and window<=0`；window>0 时 storage 收到调用方原窗口（测试锁定 7200 透传）且市场检查不执行（测试以 AssertionError 锁定）；既有测试 `test_pricing_service.py:117-135`、`test_price_fetcher_single_fetch_cache_only.py:31-83`（7200 窗口）全部通过。
- **下游 stale 标记与消费**：排除。语义路径设置 `stale=True`/`cache_status="stale_fallback"`/`source="cache_fallback"`/`is_stale`；`batch.py:82-91` stale_cache 合并、`service.py:235-236` realtime 失败后返回 stale_quote、`valuation_service.py:146-148` 按 `source=="cache_fallback" or is_stale` 计数入 `价格汇总 stale_fallback=N`，链路完整。
- **45 天/40 天窗口外条目**：排除。过期超过 40 天的条目由 storage 层拒绝返回（local_cache.py:150-151），直接 miss，与语义判定一致。
- **性能**：排除。每个过期 code 一次调用，典型 1-3 次探针迭代，40 天极端窗口实测 0.005ms。
- **生产调用方窗口=0 追踪**：排除。唯一生产 stale 调用方是 `valuation_service.py:59-72`（window 默认 0，`accept_stale_when_closed = allow_stale_price_fallback and not any_open`），即语义路径仅在三市场全闭时启用——设计意图如此。

## Residual Risk

- **锚点近似（已批准的设计取舍）**：`has_market_session_between` 以 `expires_at` 而非报价时刻为锚。临收盘 30 分钟内保存的报价，其 expiry 落在收盘后，会被语义接受，最多漏算收盘前 ≤30 分钟的价格变动（CN ≤15 分钟），且不触发 NAV 阻断告警（nav_record_service.py:57 仅阻断 价格缺失/无法获取汇率）。该取舍已在 `decisions/2026-07-22-stale-cache-semantic-acceptance.md` 第 6 条明示批准；若未来要提高精度，可在 `PriceCache` 增加报价时刻字段作为锚点。
- **stale 报价不阻断 NAV 写入**属既有行为，依赖 `价格汇总 stale_fallback=N` 与回执人工观测；语义路径降低了错误 stale 进入 final NAV 的概率，但未改变该可观测性契约。
- 节假日不在日历契约内，CN/HK 节假日会被当作交易日从而拒绝 stale（fail-closed，安全但可能造成不必要的 价格缺失 阻断）。
