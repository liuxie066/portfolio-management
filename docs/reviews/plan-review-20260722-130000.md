## Assumptions tested

1. 新浪 `gb_*` 接口能覆盖当前持仓的全部美股代码。
2. 单代码路径 `_retry_with_backoff` 对 None 结果的处理与旧 Yahoo 路径等价。
3. us_batch 中 leftover 合并新浪批量请求后，超期/异常路径不会静默丢代码或重复抓取。
4. 新浪响应解析对空串、短字段、异常数值足够健壮。
5. 缓存链路没有按 `source="yahoo_chart"` 做过滤的消费方。
6. 文档与脚本中无残留 Yahoo 引用。
7. "过期缓存兜底失效" 作为范围外项不影响本次事故修复的完整性。

## Findings

### PR-01-未修复-[高]-带点美股代码（BRK.B 为实际持仓）在新浪路径下无法解析，部署后 lx/sy 的 NAV 写入仍会被拒绝
- **位置**: `src/pricing/providers/us.py:48`、`src/pricing/providers/us_batch.py:56,68`、`src/pricing/providers/sina_us.py:118`。
- **问题类型**: 契约缺失 / 状态机漏洞 / 测试缺口
- **当前写法**: Finnhub 路径把代码做点号转换（`quote_code = code.replace(".", "-")`，us.py:34、us_batch.py:41），但新浪路径把**原始代码**直接小写拼进查询串：`"gb_" + code.lower()`（sina_us.py:118），即 `BRK.B` → `gb_brk.b`。
- **反例/失败场景**: 持仓快照中 lx 账户持有 BRK.B 25 股（富途）+20 股（长桥）、sy 账户持有 60 股（富途），合计约 35 万元市值（`.data/holdings_snapshot/lx/2026-05-24.json` snapshots[25][26]、`sy/2026-05-24.json` snapshots[13]）。`canonicalize_pricing_code` 保留内部点号（`src/pricing/classifier.py:18-20`，`tests/test_pricing_providers.py:216` 的测试名也确认 "preserves_internal_dot"），所以 `BRK.B` 原样进入新浪路径。新浪对 `gb_brk.b` 返回空串（`var hq_str_gb_brk.b="";`，计划给定的实测事实，本机复核一致）→ 解析按未命中跳过 → 批量路径落入过期缓存（us_batch.py:78-79），而过期缓存兜底因 `max_stale_after_expiry_sec=0` 的既有 bug 实际不生效（计划明确列为范围外）→ 估值时 BRK.B `quantity != 0` 且无价，产生阻断性告警 `价格缺失，无法可靠估值`（`src/app/valuation_service.py:172-173`）→ `nav_record_service.py:57,68` 拒绝 NAV 写入。也就是说：**本次事故（lx/sy NAV 写入被拒）在部署此修复后明天仍会复现**，除非 Finnhub 恰好不超时且能解析 `BRK-B`——而 Finnhub 在该主机上"大面积超时（间歇）"正是本次事故背景，不可依赖。
- **为什么有问题**: 替换兜底源时没有保持旧源对带点代码的覆盖能力（Yahoo 路径用 `BRK-B` 可以解析），且该代码是两个受影响账户的实际持仓，直接决定事故是否真正闭环。
- **直接证据**: `src/pricing/providers/sina_us.py:118`（`query = ",".join("gb_" + code.lower() for code in codes)`）；`src/pricing/providers/us.py:48`（`self.fetch_sina(code, ...)` 传原始 code，与 34 行的 `quote_code` 分叉）；`.data/holdings_snapshot/lx/2026-05-24.json`（BRK.B quantity=25.0/20.0）、`.data/holdings_snapshot/sy/2026-05-24.json`（quantity=60.0）；`src/app/nav_record_service.py:57,68`。本机实盘验证：`gb_brk.b`、`gb_brk-b`、`gb_brk_b`、`gb_brk/a` 均返回空串，而 **`gb_brk$b` 返回正常数据**（`伯克希尔B,489.6500,...`，字段结构与已核实的 7 只一致）——新浪用 `$` 表示股份类别点号。
- **影响**: 高。部署后 BRK.B 每日缺价，lx/sy NAV 写入继续被阻断性告警拒绝；手工绕过则 NAV 被低估约 35 万元。
- **建议改法和验证点**: 在 `fetch_sina_us_quotes` 中仿照 Finnhub 的 `.`→`-` 转换，查询代码做 `.`→`$` 映射（`"gb_" + code.lower().replace(".", "$")`），同时 `parse_sina_us_quotes` 的 `wanted` 映射要以转换后的查询代码为键、原始代码为值（响应 var 名回显的是 `hq_str_gb_brk$b`）。验证点：(a) 新增解析测试：`var hq_str_gb_brk$b="..."` 能命中原始代码 `BRK.B`；(b) 新增 us_batch 测试：leftover 含 `BRK.B` 时请求串为 `gb_brk$b` 且结果键为 `BRK.B`；(c) 本机/服务器 smoke：`fetch_sina_us_quotes(fetcher, ["BRK.B"])` 返回 price≈489.65。
- **修复风险（低/中/高）**: 低（纯查询串映射 + wanted 键映射，不影响其他代码路径；`$` 在 URL query 中无需编码）
- **严重程度（低/中/高/严重）**: 高

### PR-02-未修复-[中]-us_batch 的核心新行为（leftover 合并单次新浪请求、连续失败 break 路径）无测试覆盖
- **位置**: `tests/test_pricing_providers.py:163-213`；被测代码 `src/pricing/providers/us_batch.py:36-79`。
- **问题类型**: 测试缺口
- **当前写法**: 现有测试只覆盖：(a) 无 key 且全部网络失败 → 过期缓存兜底（163 行）；(b) 无 key 单代码新浪命中（186 行）；(c) 空串不触发 FX（208 行）。
- **反例/失败场景**: 本次改动最有价值也最易回归的两条路径没有任何断言：(1) **批量合并**——有 finnhub key 时 N 只代码 Finnhub 全失败后应只发**一次**新浪 HTTP 请求且结果正确分键；若未来有人改回逐代码请求或把 leftover 拼错（如重复 extend），无测试会失败。(2) **连续失败 break**——`consecutive_failures >= 3` 时 `codes[index+1:]` 并入 leftover（us_batch.py:60-64）；若切片写错（如 `codes[index:]` 导致重复、或漏 extend 导致 skipped 代码静默丢失），现有测试全部照常通过。计划第 3 步明确把"合并成一次新浪批量请求"列为改动要点，但计划第 5 步的测试更新没有对应条目。
- **为什么有问题**: 静默丢代码在此链路中等价于缺价 → NAV 拒绝，与本次事故同构；没有测试锁定 leftover 的完整性（不丢、不重）。
- **直接证据**: `git diff tests/test_pricing_providers.py` 新增测试仅两个（179、208 行）；`src/pricing/providers/us_batch.py:60-64` 的 break 切片逻辑无任何测试触达（可用 monkeypatch `fetch_finnhub` 全 raise + 计数 fake `session.get` 验证）。
- **影响**: 中。逻辑目前肉眼核对正确，但重构回归无防护。
- **建议改法和验证点**: 新增两个测试：(a) patch `_config.get` 返回 key、monkeypatch `fetch_finnhub` 抛异常，fake `session.get` 计数并返回两条 `hq_str_gb_*` 行，断言请求次数==1、URL 含 `gb_aapl,gb_futu`、两代码均命中；(b) 5 只代码、Finnhub 全失败，断言第 4、5 只未发 Finnhub 请求（计数==3）且全部经一次新浪请求命中/缓存兜底。顺带覆盖 PR-01 的 `$` 映射。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## 已排除（核实后不构成问题）

- **`_retry_with_backoff` 不重试 None**: `src/price_fetcher.py:196-198` 对 `func()` 的返回值直接 return，None 不重试。但旧 Yahoo 路径语义完全相同（空 chart 返回 None 同样不重试、不抛错），非本次改动引入的行为变化。
- **us_batch 超期/异常丢代码**: 循环顶部 `remaining_timeout`（us_batch.py:40）抛 TimeoutError 会冒泡到 `src/pricing/batch.py:156` 被捕获，随后 159-161 行统一用 stale_cache 回填未命中代码；新浪批量请求内部的 `remaining_timeout` 异常被 us_batch.py:69 捕获后逐代码走 `use_expired_cache`。无静默丢失、无重复抓取（成功代码不进 leftover）。
- **解析健壮性**: 短行/空串被 `_MIN_FIELDS=27` 与 `fields[_IDX_PRICE]` 非空检查拦截（sina_us.py:61）；`price<=0` 跳过（76 行）；`prev_close` 为 0/None 回退 price（78 行）；`change`/`change_pct` 用 `is None` 判断，0.0 不会误判（80-84 行）；GBK `errors="replace"` 与 `normalize_price_payload` 校验兜底。`wanted` 以小写代码为键、响应 var 名回显小写查询串，大小写匹配无问题。
- **缓存 source 过滤**: 全仓 grep 无任何消费方按 `yahoo_chart`/`finnhub` 等具体 source 值过滤；`data_source` 仅透传存储（`src/pricing/cache.py:59,157`、`src/local_cache.py:165,196,256`），旧 `yahoo_chart` 缓存行可正常加载。
- **脚本与配置残留**: `scripts/diagnose_pricing.py`、`SKILL.md`、`README.md`、`config.example.yaml` 均无 Yahoo 引用；`decisions/2026-05-23-pricing-phase2.md`、`CHANGELOG.md:182`、`docs/reviews/*` 中的 Yahoo 为历史记录，按 decisions 不可变日志惯例无需改写（新决策 `decisions/2026-07-22-us-pricing-sina.md` 已构成取代关系）。

- 单代码路径（`PriceFetcher.fetch`）不走 us_batch 的合并优化，逐代码一次 HTTP；与旧行为一致，可接受。
- 部署验证（计划验证步骤 3-4：服务器 smoke + 定时任务观察）尚未执行，是事故闭环的最后一环；smoke 代码清单应加入 BRK.B。

## Lens conclusions

- **Architecture boundary**: 提供层替换干净（共用 payload 构造、FX 延迟查询沿用既有设计），符号映射（`.`→`-` / `.`→`$`）应视为各 provider 的统一契约并在测试中锁定。
- **Best practice**: 失败原因打印（us_batch.py:58,62,71,78）解决了本次排查的日志静默问题，方向正确。
- **Optimal solution**: leftover 合并单次新浪请求优于旧逐代码兜底，超时预算更可控。
- **Overengineering**: 无。
- **Overcoupling**: 无新增耦合；`source="sina_us"` 无下游硬编码依赖。

## Final conclusion

`fail` until PR-01 修复并补测试（PR-02 应同批完成，PR-03 顺手修）。PR-01 是事故闭环的硬性条件：不处理 BRK.B，部署后 lx/sy 的 NAV 写入明天仍会被 `nav_record_service.py:68` 拒绝。修复方案已实盘验证（`gb_brk$b`），改动面小、风险低。
